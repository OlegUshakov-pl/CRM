# Задача: Файловое хранилище для Library (статьи + файлы)

**Контекст:** CRM на Django. В Settings уже есть настройка базовой директории для хранения проектов (`PROJECTS_ROOT` или аналог — уточни у пользователя точное имя переменной/поля). Раздел Library содержит статьи (Articles, парсенные и добавленные вручную) и отдельные файлы (Files Catalog), см. вкладки "Articles / Files / All" в разделе Library.

## Требования

### 1. Базовая папка
Создать папку `library` внутри директории, указанной в Settings (той же, где хранятся проекты).
Путь: `{PROJECTS_ROOT}/library/`

### 2. Структура для статей
При создании статьи (вручную или через парсинг) создавать отдельную подпапку:

```
library/{name}_{date}/
    {name}_{date}.md
    images/            (если есть вложенные фото)
```

- `name` — slug из заголовка статьи (транслитерация/санитайз для файловой системы)
- `date` — дата создания в формате YYYY-MM-DD
- При парсинге сайта: сохранять текст в .md, скачанные изображения — в подпапку `images/` рядом, пути в .md должны ссылаться на локальные файлы (относительные пути)

### 3. Удаление статьи
При удалении записи статьи из БД — рекурсивно удалять всю папку статьи `library/{name}_{date}/` со всем содержимым.
Логика должна быть аналогична удалению проекта в разделе Projects (посмотри, как там реализовано удаление папки проекта, и используй тот же подход/утилиту).

### 4. Отдельные файлы (Files Catalog)
Хранить прямо в `library/` (без создания вложенной папки на каждый файл, если это единичный файл без вложений).
При удалении записи файла — удалять соответствующий физический файл с диска.

## Важное

- Использовать одну и ту же функцию/утилиту удаления папки для Library, что и для Projects (не дублировать логику).
- При удалении — обернуть в try/except, чтобы ошибка удаления файла с диска не ломала транзакцию удаления записи из БД (или наоборот, в зависимости от того, что важнее — консистентность БД или файлов).
- Санитайзить имена файлов/папок от недопустимых символов (кириллица, пробелы, спецсимволы из заголовков статей).
- Учесть, что путь к директории берётся динамически из Settings, а не хардкодится.

## Что проверить после реализации

- [ ] создание статьи вручную → папка создаётся в нужном месте
- [ ] парсинг статьи с сайта → .md + картинки сохраняются в свою подпапку
- [ ] удаление статьи → папка полностью пропадает с диска
- [ ] удаление отдельного файла → файл пропадает, папка library остаётся
